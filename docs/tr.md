# Техническое решение проекта «Distributed Messenger»

## Введение

**Цель проекта:**  
Создание распределенного мессенджера с микросервисной архитектурой.

**Основания для разработки:**  
Учебный проект по изучению распределенных систем

## Глоссарий

| Термин | Определение |
|--------|-------------|
| **Пользователь** | Физическое лицо, использующее мессенджер для общения |
| **Зарегистрированный пользователь** | Пользователь, прошедший процедуру регистрации и имеющий учетную запись в системе |
| **Друг** | Пользователь, добавленный в список контактов с взаимным подтверждением |
| **Сообщение** | Текстовые данные, отправляемые одним пользователем другому |
| **Чат** | Виртуальное пространство для обмена сообщениями между пользователями |
| **Профиль** | Публичная информация о пользователе: имя, аватар, статус |
| **Запрос в друзья** | Приглашение одного пользователя другому для добавления в контакты |
| **Аутентификация** | Процесс проверки подлинности пользователя при входе в систему |
| **Распределенная система** | Система, компоненты которой расположены на разных сетевых узлах |

## Функциональные требования

### 1. Регистрация и аутентификация пользователей
**Регистрация по email:**
- Ввод email, логина и пароля при регистрации
- Проверка уникальности email и логина
- Отправка email с уникальной ссылкой активации
- Подтверждение регистрации по ссылке из email
- Автоматическое удаление неподтвержденных аккаунтов через 24 часа

**Аутентификация:**
- Вход по email и паролю
- Генерация JWT токенов при успешной аутентификации
- Восстановление пароля через email с временной ссылкой
- Защита от brute-force атак (ограничение попыток входа)

**Планируемые расширения:**
- Аутентификация через OAuth (Google, GitHub)
- Регистрация по номеру телефона с SMS подтверждением
- Двухфакторная аутентификация

### 2. Управление профилем пользователя
- Создание и редактирование профиля
- Загрузка и обновление аватара
- Установка статуса (онлайн, офлайн, занят)
- Настройки приватности профиля

### 3. Поиск и добавление в друзья
- Поиск пользователей по email или логину
- Отправка запросов на добавление в друзья
- Принятие или отклонение входящих запросов
- Просмотр списка друзей и входящих/исходящих запросов

### 4. Обмен текстовыми сообщениями в реальном времени
- Отправка текстовых сообщений друзьям
- Доставка сообщений через WebSocket соединения
- Сохранение истории сообщений
- Уведомления о новых сообщениях
- Доставка офлайн-сообщений при подключении пользователя

### 5. История сообщений
- Хранение полной истории переписки
- Загрузка истории при открытии чата
- Поддержка пагинации для длинных переписок

## Нефункциональные требования

- **Архитектура:** Микросервисная архитектура с API Gateway и отдельными БД для каждого сервиса
- **Отказоустойчивость:** Система сохраняет работоспособность при отказе отдельных компонентов
- **Безопасность:** 
  - Защита от SQL-инъекций и XSS
  - Шифрование передаваемых данных (TLS)
  - Хранение паролей в хешированном виде
  - JWT токены с ограниченным временем жизни
- **Масштабируемость:** Возможность горизонтального масштабирования отдельных сервисов

## Пользовательские сценарии

### 1. Регистрация нового пользователя

**Предусловия:** Пользователь не зарегистрирован в системе

**Основной поток:**
1. Пользователь вводит email, логин и пароль
2. Система проверяет уникальность данных
3. Создается учетная запись и профиль пользователя
4. Система отправляет email со ссылкой для подтверждения регистрации
5. Пользователь переходит по ссылке для активации аккаунта
6. После активации пользователь получает доступ к системе

**Альтернативный поток:**
- Если email уже зарегистрирован - система сообщает об ошибке
- Если ссылка подтверждения не активирована в течение 24 часов - аккаунт удаляется

### 2. Добавление в друзья и управление контактами
 
**Предусловия:** Пользователь авторизован в системе

**Основной поток:**
1. Пользователь ищет другого пользователя по логину/email
2. Система возвращает результаты поиска
3. Пользователь отправляет запрос на добавление в друзья
4. Получатель уведомляется о запросе
5. Получатель принимает/отклоняет запрос
6. При принятии - пользователи добавляются в друзья друг другу

**Альтернативный поток:**
- Если пользователь не найден - система сообщает об ошибке
- Если запрос уже отправлен - система уведомляет об этом

### 3. Обмен сообщениями в реальном времени

**Предусловия:** Оба пользователя авторизованы и являются друзьями

**Основной поток:**
1. Пользователь выбирает собеседника из списка друзей
2. Отправляет текстовое сообщение
3. Система доставляет сообщение получателю если он онлайн
4. Если получатель офлайн - сообщение сохраняется и доставляется при следующем входе
5. Оба пользователя видят историю переписки

**Альтернативный поток:**
- Если пользователь не в друзьях - отправка сообщения блокируется
- При ошибке доставки система предпринимает повторные попытки

## Технические сценарии

### Технический сценарий 1: Регистрация пользователя

**Предусловия:**
- Все микросервисы доступны и функционируют нормально
- Базы данных доступны для записи
- Email сервис настроен и готов к отправке писем

**Основной поток:**
1. Клиент отправляет POST запрос на `/api/register` с данными регистрации
2. API Gateway перенаправляет запрос в Auth Service
3. Auth Service валидирует входные данные и проверяет уникальность email/логина
4. Auth Service создает запись пользователя в базе данных аутентификации
5. Auth Service вызывает User Service для создания профиля пользователя
6. Auth Service генерирует токен активации и вызывает Notification Service
7. Notification Service отправляет email с ссылкой активации
8. Auth Service возвращает клиенту подтверждение о отправке письма
9. При переходе по ссылке активации Auth Service активирует учетную запись

**Альтернативный поток A: Ошибка валидации**
- На шаге 3: Если данные невалидны или email/логин уже существуют
- Auth Service возвращает ошибку 400 с описанием проблемы
- Процесс регистрации прерывается

**Альтернативный поток B: Ошибка отправки email**
- На шаге 7: Если отправка email не удалась
- Auth Service помечает пользователя как "требующий повторной отправки"
- Система предпринимает повторные попытки отправки

### Технический сценарий 2: Отправка сообщения

**Предусловия:**
- Оба пользователя зарегистрированы и авторизованы
- Пользователи являются друзьями
- WebSocket соединение установлено для онлайн-пользователей

**Основной поток:**
1. Клиент отправляет POST запрос на `/api/messages` с JWT токеном и данными сообщения
2. API Gateway валидирует JWT через Auth Service
3. API Gateway перенаправляет запрос в Chat Service
4. Chat Service проверяет через Friend Service, являются ли пользователи друзьями
5. После подтверждения дружбы Chat Service сохраняет сообщение в MongoDB
6. Chat Service уведомляет Notification Service о новом сообщении
7. Notification Service доставляет сообщение получателю через WebSocket
8. Chat Service возвращает подтверждение отправки отправителю

**Альтернативный поток A: Пользователи не друзья**
- На шаге 4: Friend Service возвращает отрицательный результат
- Chat Service возвращает ошибку 403 "Пользователи не являются друзьями"
- Сообщение не сохраняется и не доставляется

**Альтернативный поток B: Получатель офлайн**
- На шаге 7: Если WebSocket соединение с получателем отсутствует
- Notification Service помечает сообщение как "ожидающее доставки"
- Сообщение доставляется при следующем подключении получателя

### Технический сценарий 3: Добавление в друзья

**Предусловия:**
- Оба пользователя зарегистрированы и авторизованы
- Пользователи не являются друзьями и не имеют активных запросов

**Основной поток:**
1. Клиент отправляет POST запрос на `/api/friend-requests` с ID целевого пользователя
2. API Gateway валидирует JWT и перенаправляет запрос в Friend Service
3. Friend Service проверяет существование целевого пользователя через User Service
4. Friend Service создает запрос в друзья со статусом "pending" в PostgreSQL
5. Friend Service уведомляет Notification Service о новом запросе
6. Notification Service доставляет уведомление целевому пользователю через WebSocket
7. Friend Service возвращает подтверждение создания запроса

**Альтернативный поток A: Запрос уже существует**
- На шаге 4: Если запрос между пользователями уже существует
- Friend Service возвращает ошибку 409 "Запрос уже отправлен"
- Новый запрос не создается

**Альтернативный поток B: Пользователи уже друзья**
- На шаге 3: Если пользователи уже являются друзьями
- Friend Service возвращает ошибку 409 "Пользователи уже друзья"
- Запрос не создается
