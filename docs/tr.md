## Введение

**Цель проекта:**  
Создание распределенного мессенджера с микросервисной архитектурой.

## Глоссарий

| Термин | Определение |
|--------|-------------|
| **Пользователь** | Физическое лицо, использующее мессенджер для общения |
| **Зарегистрированный пользователь** | Пользователь, прошедший процедуру регистрации и имеющий учетную запись в системе |
| **Друг** | Пользователь, добавленный в список контактов с взаимным подтверждением |
| **Сообщение** | Текстовые данные, отправляемые одним пользователем другому |
| **Чат** | Виртуальное пространство для обмена сообщениями между пользователями |
| **Профиль** | Публичная информация о пользователе: имя, аватар, статус |
| **Запрос в друзья** | Приглашение одного пользователя другому для добавления в контакты |
| **Аутентификация** | Процесс проверки подлинности пользователя при входе в систему |
| **Распределенная система** | Система, компоненты которой расположены на разных сетевых узлах |

---

## Функциональные требования

Система должна предоставлять следующие функции:

1. **Регистрация и аутентификация пользователей**
2. **Управление профилем пользователя**
3. **Поиск и добавление в друзья**
4. **Обмен текстовыми сообщениями в реальном времени**
5. **История сообщений**

---

## Нефункциональные требования

- **Архитектура:** Проект будет реализован с помощью микросервисной архитектуры(API Gateway, для каждого микросервиса будет своя БД)
- **Отказоустойчивость:** Система должна сохранять работоспособность при отказе отдельных компонентов
- **Безопасность:** Шифрование передаваемых данных, защита от основных уязвимостей

## Пользовательские сценарии

### 1. Регистрация нового пользователя

**Предусловия:** Пользователь не зарегистрирован в системе

**Основной поток:**
1. Пользователь вводит email, логин и пароль
2. Система проверяет уникальность данных
3. Создается учетная запись и профиль пользователя
4. Отправляется email для подтверждения регистрации
5. Пользователь получает доступ к системе

### 2. Добавление в друзья и управление контактами
 
**Предусловия:** Пользователь авторизован в системе

**Основной поток:**
1. Пользователь ищет другого пользователя по логину/email
2. Система возвращает результаты поиска
3. Пользователь отправляет запрос на добавление в друзья
4. Получатель уведомляется о запросе
5. Получатель принимает/отклоняет запрос
6. При принятии - пользователи добавляются в друзья друг другу

### 3. Обмен сообщениями в реальном времени

**Предусловия:** Оба пользователя авторизованы и являются друзьями

**Основной поток:**
1. Пользователь выбирает собеседника из списка друзей
2. Отправляет текстовое сообщение
3. Система доставляет сообщение получателю если он онлайн
4. Если получатель офлайн - сообщение сохраняется и доставляется при следующем входе
5. Оба пользователя видят историю переписки


##  Архитектура системы

- **API Gateway** - единая точка входа, маршрутизация, аутентификация
- **Auth Service** - регистрация, аутентификация, JWT токены
- **User Service** - управление профилями, поиск пользователей
- **Friend Service** - управление друзьями, запросы на добавление
- **Chat Service** - обмен сообщениями, история чатов

## Технические сценарии

### Сценарий 1: Регистрация пользователя
Клиент → API Gateway → Auth Service

Auth Service → User Service (создание профиля)

Auth Service → Notification Service (email подтверждение)

Возврат JWT токена клиенту


### Сценарий 2: Отправка сообщения

Клиент -> API Gateway (с JWT) -> Chat Service

Chat Service -> Friend Service (проверка дружбы)

Chat Service -> MongoDB (сохранение сообщения)

Chat Service -> Notification Service (доставка через WebSocket)


### Сценарий 3: Добавление в друзья

Клиент -> API Gateway -> Friend Service

Friend Service -> User Service (валидация пользователя)

Friend Service -> Notification Service (уведомление)

Friend Service -> PostgreSQL (сохранение запроса)
